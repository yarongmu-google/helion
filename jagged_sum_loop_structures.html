Helion source — loops in N-space (item-axis)
# Iterating over items, not rows. Each tile_b carries `tile_b.block_size` items.
for tile_b in hl.tile(N):              # outer: pick items     →  block_size = 2  →  2 iters (items {0,1}, {2,3})
  starts = x_offsets[tile_b]            #  e.g. [ 0,  3]
  ends   = x_offsets[tile_b.index+1]
  nnz    = ends - starts             #  e.g. [ 3,  7]   ← per-item lengths

  for tile_m in hl.tile(M):            # middle: pick cols     →  block_size = 2  →  4 iters
    row_sums = hl.zeros([tile_b, tile_m])

    for tile_k in hl.jagged_tile(nnz): # inner: jagged rows    →  block_size = 4  →  ceil(amax(nnz)/4) iters
      x_slice = hl.load(x_flat, [flat_indices])
      row_sums += x_slice.sum(dim=1)

    out[tile_b, tile_m] = row_sums
4/4 masked, item 1 → 1/4 Each tile_b has its own amax → per-tile_b padding. All iterations (this kernel): tile_b ∈ {0, 1} × tile_m ∈ {0..3} = 2 × 4 = 8 outer-program-iters total. Each walks tile_k up to its own amax. Shown: tile_b=0, tile_m=0, tile_k=0 items {0, 1}, cols {0, 1}, rows {0..3} per item
TPU/Pallas — grid in L-space (flat-row-axis)
# Pallas grid sees flat L, not items. Item boundaries come from offsets.
def kernel(x_ref, offsets_ref, out_ref):
  i_L, i_M = pl.program_id(0), pl.program_id(1)   # grid: (3, 4)
  l_start = i_L * block_L                            # block_L=8, l_start ∈ {0,8,16}

  # x_ref is VMEM slab [block_L=8, block_M=2] — DMA'd by BlockSpec from HBM.
  # Walk all items, accumulate into out_ref for each one this block touches.
  for item_idx in range(N):                          # static loop, unrolled
    it_start = offsets_ref[item_idx]
    it_end   = offsets_ref[item_idx+1]
    lo = jnp.maximum(it_start, l_start) - l_start
    hi = jnp.minimum(it_end,   l_start+block_L) - l_start
    if lo < hi:                                       # item has rows in this L-block
      partial = x_ref[pl.ds(lo, hi-lo), :].sum(axis=0)
      out_ref[item_idx, :] += partial

The design: per-seq dynamic loop with double-buffered DMA

Single-program grid; pl.loop iterates over sequences inside the kernel. For each sequence, bx_sz-row blocks of input are fetched from HBM into a double-buffered VMEM scratch via pltpu.make_async_copy; each block is reduced and accumulated into an fp32 scratch row; at the sequence boundary the accumulator is cast, copied to a separate double-buffered output VMEM slot, and DMA'd back to HBM. The assembly is the application of building blocks N1–N8 (see panel below); the kernel body is ~80 lines minus comments.

Running example used throughout the walkthrough: B = 4, ni = [3, 7, 5, 9], L = 24, M = 8, x_offsets = [0, 3, 10, 15, 24], fetch block size bx_sz = 8. After lane-padding, M_padded = 128; x is (24, 128) (zero-padded along the lane dim — zero is the sum identity), and out is (4, 128) in HBM, trimmed back to (4, 8) by prepare_outputs.

Detailed view of iter 4 → iter 5 (seq 3, split across two bx blocks — the only multi-block seq in this example):
  1. Iter 4 (seq=3, bx_idx=0): bx_sem=1 → consume buf 1. @pl.when(bx_idx == 0) fires (N8): acc_ref[...] = 0. get_next_bx_ids returns (seq=3, bx=1, sem=0): still inside seq 3, advance to bx_idx=1, flip ping-pong. Prefetch starts fetching the second bx of seq 3 into buf 0; SMEM[0] is committed to 0 first so the next iter inherits the new state (N7). Wait on buf 1; buf 1 holds rows 15..22 (8 rows, fully valid). Reduce: bx_block.sum(axis=0) over all 8 rows; acc_ref += sum. No flush — bx_idx=0 ≠ num_bx-1=1.
  2. Iter 5 (seq=3, bx_idx=1): bx_sem=0 → consume buf 0. @pl.when(bx_idx == 0) does NOT fire — acc_ref still holds the partial sum from iter 4. get_next_bx_ids returns (seq=4, bx=0, sem=1); next_seq ≥ num_seqs → prefetch guard suppresses the start_fetch. Wait on buf 0; buf 0 holds row 23 plus 7 stale lanes. Mask (iota < sz=1) zeros out rows 1..7; bx_masked.sum(axis=0) contributes just row 23 to acc. @pl.when(bx_idx == num_bx-1) fires (N8): bo_sem=1, flip SMEM[1] to 0, wait_send_bo(1) drains seq-1's bo (which had been in flight since iter 2). Cast acc to out_dtype, store to bo_x2_ref[1], start the bo DMA to out[3, :]; record bo_ids_ref[1] = 3.
  3. Epilogue (still inside iter 5, since seq_idx == num_seqs - 1 == 3): for i in range(2): wait_send_bo(i) drains the two remaining in-flight bo DMAs — buf 0 (seq 2) and buf 1 (seq 3 we just started). After this the kernel returns and HBM out is complete.
Building blocks applied in this design
Cross-reference to the inherited primitives panel below.
  • N1 online accumulator — read prev / combine / write back — the per-block acc_ref[...] = acc_ref[...] + bx_masked.sum(axis=0). Same machinery as RPA's softmax stats; simpler math (plain add).
  • N2 broadcast_minor — not used by plain jagged_sum. Becomes relevant for jagged_mean (divide acc by ni broadcast across M lanes at flush).
  • N3 _async_copy — verbatim, called from _fetch_bx and _send_bo.
  • N4 degenerate-copy semaphore wait — used in the wait=True branch of _fetch_bx and _send_bo; lets the wait happen in a different closure from where the original DMA started.
  • N5 re-derive seq quantities inside closures — _derive_iter_quantities(seq_idx, cu_x_lens_ref) called locally inside _fetch_bx so the prefetch look-ahead (called with next seq_idx) sees the right start/end.
  • N6 per-buffer SMEM payload (bo_ids_ref[2]) — start_send_bo(seq_idx, ...) writes seq_idx to bo_ids_ref[bo_sem_idx]; wait_send_bo(bo_sem_idx) reads it back. @pl.when(0 ≤ old_seq_idx ≤ seq_idx) guard skips the cold-start case (sentinel -1).
  • N7 SMEM ping-pong rotation (sem_ids_ref[2]) — index 0 = bx, 1 = bo. Each iter reads at top, computes flipped next via lax.select, writes flipped value to SMEM inside prefetch_next_bx/flush_bo.
  • N8 seq-boundary acc-init and bo-flush — @pl.when(bx_idx == 0) guards acc init; @pl.when(bx_idx == num_bx - 1) guards bo flush. Without these guards we'd zero acc per block (wrong: drops partial sums in multi-block seqs like seq 3) and flush per block (wasted DMAs + wrong output for multi-block seqs).
Constraints and trade-offs
  • Single-program grid (grid=(1,)). All seqs processed sequentially inside one program by the inner pl.loop. Not grid-parallel over seqs — parallelism is over Megacore cores at a higher level (RPA notes this as TODO; same applies to us).
  • bx_sz must be a multiple of 8 (sublane alignment). Tunable; default 64 in the public API. Larger = fewer DMA round-trips per long seq, more VMEM, more masked-out compute on short partial-tail blocks. Bounded by vmem_limit_bytes.
  • M_padded must be a multiple of 128 (lane alignment). prepare_inputs zero-pads if M_actual < M_padded; prepare_outputs trims back. Zero is the sum identity so padded lanes contribute nothing — for jagged_mean the padded lanes also divide by ni harmlessly.
  • Partial-tail block masking (when sz < bx_sz): unfetched VMEM rows hold stale data from prior DMAs, so we mask with iota < sz and zero-fill before the reduce. Static-shape — Mosaic-friendly.
  • Strategy-3 hook preserved: static_q_len kwarg on _jagged_sum_kernel_loop and a dual-branch inside process (Strategy 2 derives num_bx from dynamic seq_len; Strategy 3 from static_q_len). Wiring the host-side workitem arrays (starts, ends, out_rows, modes) into scalar_prefetches is the remaining work to enable Strategy 3.
Inherited building blocks (the distilled design above is their assembly)
Reusable implementation primitives from tpu-inference/.../ragged_paged_attention/v3/kernel.py that map 1:1 to the data-movement and accumulation scaffolding the design needs. Identified from line-by-line walkthrough; each is production-tested in RPA.
  • Online accumulator update pattern. RPA reads previous VMEM-scratch state, combines with the new block's contribution, writes back. RPA's flavour is FlashAttention's online softmax with running (m_prev, l_prev) stats and rescale-by-exp(m_prev − m_curr). For jagged_sum it collapses to acc_ref[...] += block_sum on an fp32 scratch buffer, initialized at each new sequence and flushed at sequence end. Same machinery, simpler math.
  • _async_copy(src, dst, sem, wait) helper. 9-line wrapper around pltpu.make_async_copy that takes a wait flag. Callers write start_fetch_*(...) (wait=False) and wait_fetch_*(...) (wait=True) as thin one-liners. Reuse verbatim.
  • Degenerate-copy semaphore wait. _async_copy(src=dst, dst=dst, sem, wait=True) waits on a prior DMA's semaphore when the original copy object isn't in scope (start and wait live in different closures). Hardware sees the semaphore in flight and waits; the redundant self-copy is collapsed because src and dst alias. Canonical Pallas idiom.
  • Re-derive seq quantities inside fetch/send closures. Closures take seq_idx as a parameter and call _derive_iter_quantities locally rather than capturing seq bounds from the outer scope. Required because the closures are invoked with different seq_idx values — current sequence AND a look-ahead "next sequence to prefetch." Capturing would pin the closure to the outer call's seq_idx and break double-buffered prefetch.
  • broadcast_minor(scalar, shape) helper. Broadcasts a per-row scalar across the lane (M) dim for elementwise ops. Used by RPA for exp(s − m_curr) in softmax. For jagged_sum not strictly needed (plain vector add). For jagged_mean, used to broadcast the per-row divisor ni across M lanes for the final divide. Lift when we extend beyond plain sum.
  • Per-buffer SMEM bookkeeping for split start/wait. Ping-pong VMEM buffers don't carry metadata; the "what work is this buffer currently holding" record lives in a tiny SMEM scratch (RPA: bo_ids_ref[2 × num_buffers] recording (seq_idx, block_idx) per buffer). When start_send_* is called it writes the record; when wait_send_* is called with only the buffer index, it reads the record back to know what to wait on. A @pl.when(...) guard skips the wait at cold start (buffer not yet used) and after a prior wait disarms the guard (RPA's "set size = 0 after wait" trick). For jagged_sum: bo_ids_ref with one seq_idx per buffer (2 entries for double buffering) is enough — the block dim is implicit because output is one row per seq.
  • SMEM-coordinated ping-pong rotation. Distinct from N6 (which records per-buffer payload) — this is about which buffer slot is current. Each fetch type (bq, bkv, bo in RPA; bx and bo for us) gets one i32 slot in sem_ids_ref recording the current ping-pong index. Each iteration: (a) reads the current index from SMEM at the top (bq_sem_idx = sem_ids_ref[0]); (b) computes the flipped next index via the get_next_*_ids helper (next_bq_sem_idx = lax.select(bq_sem_idx == 0, 1, 0)); (c) when prefetching, writes the flipped index to SMEM AND starts the fetch (order within the iter doesn't matter — both must happen before the next iter starts); (d) uses the un-flipped index for the current iter's wait and compute. The next iteration inherits the flipped value through SMEM, with no need to pass any state through the @pl.loop body closure. Canonical Pallas ping-pong rotation idiom.
  • Seq-boundary acc-init and bo-flush (vs RPA's per-bq-block). Structural difference, not a reusable RPA mechanism — but important to call out because it determines where guards go in our analog of compute_with_bq. In RPA, each bq block produces its own output rows, so acc_ref[...] = 0 happens at the START of every bq iteration (line 1301) and start_send_bo(...) fires at the END of every bq iteration (line 1457). In jagged_sum, all bx blocks of one sequence accumulate into the SAME one-row output, so the acc init must move to the FIRST bx of each seq and the bo flush must move to the LAST bx of each seq. Implementation: @pl.when(bx_idx == 0) def init_acc(): acc_ref[...] = 0 and @pl.when(bx_idx == num_bx − 1) def flush_bo(): ... start_send_bo(...). Same machinery as RPA's epilogue; different guard predicates. Per-iter behavior of the output-side DMA also differs: RPA fires one DMA per bq block (output is jagged like input); jagged_sum fires one DMA per seq (output is dense, B rows).
