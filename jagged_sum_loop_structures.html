Helion source — loops in N-space (item-axis)
# Iterating over items, not rows. Each tile_b carries `tile_b.block_size` items.
for tile_b in hl.tile(N):              # outer: pick items     →  block_size = 2  →  2 iters (items {0,1}, {2,3})
  starts = x_offsets[tile_b]            #  e.g. [ 0,  3]
  ends   = x_offsets[tile_b.index+1]
  nnz    = ends - starts             #  e.g. [ 3,  7]   ← per-item lengths

  for tile_m in hl.tile(M):            # middle: pick cols     →  block_size = 2  →  4 iters
    row_sums = hl.zeros([tile_b, tile_m])

    for tile_k in hl.jagged_tile(nnz): # inner: jagged rows    →  block_size = 4  →  ceil(amax(nnz)/4) iters
      x_slice = hl.load(x_flat, [flat_indices])
      row_sums += x_slice.sum(dim=1)

    out[tile_b, tile_m] = row_sums
4/4 masked, item 1 → 1/4 Each tile_b has its own amax → per-tile_b padding. All iterations (this kernel): tile_b ∈ {0, 1} × tile_m ∈ {0..3} = 2 × 4 = 8 outer-program-iters total. Each walks tile_k up to its own amax. Shown: tile_b=0, tile_m=0, tile_k=0 items {0, 1}, cols {0, 1}, rows {0..3} per item
TPU/Pallas — grid in L-space (flat-row-axis)
# Pallas grid sees flat L, not items. Item boundaries come from offsets.
def kernel(x_ref, offsets_ref, out_ref):
  i_L, i_M = pl.program_id(0), pl.program_id(1)   # grid: (3, 4)
  l_start = i_L * block_L                            # block_L=8, l_start ∈ {0,8,16}

  # x_ref is VMEM slab [block_L=8, block_M=2] — DMA'd by BlockSpec from HBM.
  # Walk all items, accumulate into out_ref for each one this block touches.
  for item_idx in range(N):                          # static loop, unrolled
    it_start = offsets_ref[item_idx]
    it_end   = offsets_ref[item_idx+1]
    lo = jnp.maximum(it_start, l_start) - l_start
    hi = jnp.minimum(it_end,   l_start+block_L) - l_start
    if lo < hi:                                       # item has rows in this L-block
      partial = x_ref[pl.ds(lo, hi-lo), :].sum(axis=0)
      out_ref[item_idx, :] += partial

Three lowering strategies — illustrated on the running example

Running example throughout this section: N = 4, ni = [3, 7, 5, 9], L = 24, M = 8, x_offsets = [0, 3, 10, 15, 24], Pallas BLOCK_L = 8, BLOCK_M = 2. Per-item L-ranges: item 0 → L[0,3), item 1 → L[3,10) (crosses L-block boundary at L=8), item 2 → L[10,15), item 3 → L[15,24) (crosses L-block boundary at L=16).

Strategy (1) — SMEM-side preprocessing
Compiler reframes the kernel to walk L-blocks. x_offsets stays on the host but its per-block decomposition is computed inside the kernel by a one-time SMEM scalar scan before the VMEM pipeline starts.

The * problem. Without the synthetic entries the stream would be [(0,0), (0,3), (1,2), (1,7), (2,8)]. Item 1's contribution shows up as the pair (0,3) → (1,2) — a range whose start and end are in different L-blocks. Mosaic can't address such a slice from a single VMEM tile; the per-block compute can't decode it. The fix is to cut the row at the block boundary by inserting synthetic (prev_blk, BLOCK_L) entries every time the block index jumps. After insertion the stream becomes [(0,0), (0,3), (0,8)*, (1,2), (1,7), (1,8)*, (2,8)]. Now every consecutive pair lies in one block, and items spanning blocks are correctly represented as two contributions (head + tail) that the VMEM scratch accumulator re-stitches.

For this example the * problem inserts 2 extra entries (5 → 7). For long rows where ni > 2·BLOCK_L the insert is a while-loop (one synthetic per crossed boundary), and the SMEM scratch sizing must reserve for ceil(max(ni) / BLOCK_L) − 1 inserts per row.

Pros
  • Input stable: x_offsets is given on the host, fully known before launch. No host preprocessing call needed; everything self-contained in the kernel.
  • Single wide aligned VMEM load per L-block program; high arithmetic intensity per DMA.
  • Chunking handles oversized items naturally (a long row simply spans more L-blocks; no special case).
Cons
  • The * problem: boundary-crossing insertion logic. For long rows (ni > 2·BLOCK_L) it becomes a while-loop inside the scan, with SMEM scratch sized for the worst case.
  • SMEM/VMEM pipeline gap: the SMEM scan runs serially before the VMEM pipeline starts (it has to — the pipeline needs block_start[] to know how many contributions per block). For B in the 100–1000 range, that's 100–1000+ SMEM cycles of pure overhead during which VMEM is idle. That's several matmul-cycles' worth of wasted time per kernel call.
  • Per-block compute waste bounded by (BLOCK_L − avg_ni_in_block) / BLOCK_L — full-width masked reduce even when the row only covers part of the block.
Strategy (2) — Dynamic loop (per-item async copy)
No L-space reframe. Compiler keeps the source's for tile_b structure. x_offsets goes into SMEM via scalar prefetch; each iteration reads its (start, end) from SMEM and asynchronously copies the per-item slice of x_data into VMEM scratch.

Mosaic's alignment check is on the destination VMEM ref (always aligned by allocation), not on the HBM source offset — so x_data[start:end, :] with a runtime start is fine. After all iterations a single DMA copies o_vmem → out.
When one item exceeds VMEM (m_tile / k_tile swap)
Suppose item 3 had n3 = 10,000 — that won't fit in VMEM at full M width. The fix: chunk along the row axis inside the iteration. If the source kernel has for tile_m: for tile_k:, the compiler must swap to for tile_k: for tile_m: so tile_k chunks the per-item rows in multiple async_copies. That's a constrained loop-interchange in the lowering pass — well-defined but it's a real change to source-loop order.
Pros
  • Easy to lower: the compiler injects scalar prefetch + async_copy + VMEM scratch around the existing per-sequence kernel body. The body itself is unchanged from the canonical Helion source.
  • Sticks to what the user wrote: the for tile_b loop in the source maps directly to the runtime loop in the kernel. No N-space → L-space reframe to reason about.
  • No SMEM scan overhead — x_offsets reads are amortized into the per-iteration hot loop (2 scalar reads each).
Cons
  • Potential VMEM waste: scratch sized to max(ni) × M. Bounded by max / avg. For our example the smallest item uses 3/9 sublanes (67% idle); a kernel with one outlier item drags down all the rest.
  • Tile-loop reorder when one item exceeds VMEM: the lowering must swap tile_m with tile_k so tile_k can chunk per-item rows. Requires the source kernel to expose tile_k as a chunk-able axis.
  • One async_copy per item — sequential, no cross-item pipelining. DMA setup overhead amortizes badly for very short items.
Strategy (3) — Helion source rewrite + host-side preprocessing on CPU
User opts in via a new Helion primitive (e.g. hl.jagged_workitems(x_offsets)). The Helion launcher does the preprocessing on the host CPU before kernel launch: flatten the per-item work into a uniform list of (start, end, output_row, mode) work-items, pre-chunking any oversized items. The kernel iterates this list linearly — no tile_k, no loop-interchange.

Each kernel iteration reads (start, end, out_row, mode) from SMEM, does one async_copy, reduces, then writes-or-accumulates into o_vmem[out_row, :]. The kernel has no concept of "item" or "chunk" — just a flat work-item index k = 0..K−1. The host pre-decided what each iteration does.
Pros
  • Same as Strategy (2): easy to lower (per-iteration async_copy around a small kernel body); no SMEM scan blocking the VMEM pipeline; alignment falls out of VMEM scratch sizing.
  • No tile_k needed — and no loop-interchange. Oversized items are chunked on the host before launch; the kernel just sees more work-items in the flat list.
  • VMEM scratch sizing is deterministic (compile-time chunk budget) instead of max(ni)-dependent. No VMEM waste from outlier items.
  • Opt-in via a Helion primitive — no compiler pattern-matching, no risk of mis-rewriting unrelated kernels.
Cons
  • Source rewrite required: user must change their kernel to use the new primitive. Less "magic" than (1) or (2).
  • Adds a small per-launch host CPU cost (typically negligible for jagged kernels — microseconds — but real).
  • Triton/GPU lowering of the same primitive needs a fallback (likely a no-op rewrite back to the canonical gather form, since GPU doesn't need this dance).
